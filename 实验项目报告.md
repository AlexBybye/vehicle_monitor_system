## 一、项目概述

车流监控系统（Vehicle Monitor System，VMS）是一个**离线可用的城市区域车流可视化前端应用**。它对接本地模拟服务端（`http://127.0.0.1:12345`），在一张 800×600 的区域地图上实时展示出入口、检查站、道路网络与行驶车辆，并提供历史查询、轨迹回放、统计分析与路径拥堵分析四大功能页。

项目最大的工程特点是**全链路 Mock 兜底**：每一个接口在请求失败时都会回退到内置的、与 API 文档字段完全一致的模拟数据。因此即便在没有 Windows 模拟服务端的 macOS / Linux 环境下，所有功能仍可完整演示——这是本项目能够脱离后端独立运行的关键。

**技术栈（实际使用）：**

| 层次 | 选型 |
| --- | --- |
| 框架 | Vue 3.5（`<script setup>` 组合式 API） |
| 构建 | Vite 8 |
| 语言 | TypeScript 6 |
| 状态管理 | Pinia 3（单一 store） |
| 路由 | vue-router 5（`createWebHistory`） |
| 渲染 | 原生 Canvas 2D，双层（静态层 + 动态层）+ `requestAnimationFrame` 插值；地图、图表均手写绘制，**未引入任何图表/地图库** |
| 测试 | Vitest 4 + jsdom（仅有脚手架默认用例） |

> 说明：设计文档提到的 ECharts 未使用——地图与统计图表全部为**纯手写 Canvas 实现**，无第三方图表/地图库依赖。双层 Canvas、`requestAnimationFrame` 插值、Service Worker 离线缓存等在本次迭代中已全部落地（详见第二、五节）。

### 2.2 ⭐ 我额外实现的加分功能（重点）

以下功能**均经过源码核实确实可运行**，是本项目相对基础要求的增量价值：

**① 全链路离线 Mock 兜底（无后端可演示全部功能）**
每个接口 `try/catch` 失败即回退内置模拟数据。`getMockHistoryFor()` 用**车牌号哈希做种子的伪随机数生成器**为每辆车生成稳定、可复现的多样化历史（不同入口/出口/时间/费用），并按车牌缓存，避免每次刷新数据漂移。这让"平均车速""停留时间""过去一小时分布"等统计在纯前端也能被真实验证。

**② 车辆历史轨迹回放（Dijkstra 最短路 + BFS 全检查站环路）**
历史记录的"📍 轨迹"按钮会在与地图一致的**有向道路图**上计算路线并在 Canvas 上绘制：
- 入口≠出口：`dijkstra()` 求最短路径；
- 入口==出口（环绕）：`createCircularPath()` 用**带访问位掩码的 BFS**求"经过全部 6 个检查站再回到起点"的最短闭合回路（禁止原地折返）；
- 再对节点序列按距离密度插值成连续折线，并标注起点/终点。

**③ 路径拥堵分级 + 卡口压力预警**
`calculatePathCongestion()` 把整张路网按 100px 切分为 segment，用**点到线段距离 < 30px** 判定车辆归属，按端点对聚合计数并分级（0 畅通 / 1-3 轻度 / 4-6 中度 / ≥7 严重）。`triggerCongestionAlerts()` 对达到严重级的主路径触发预警。拥堵分析页提供"全部/畅通/拥堵/严重"过滤标签与等级进度条。

**④ 累计统计（基于"位置边沿检测"的事件计数）**
为解决"车辆离开后统计归零"的问题，store 维护了 `knownVehicleNos`（曾出现过的全部车牌，永不删除）、`cumulativeCheckpointPasses`（每辆车从"不在检查点范围"→"进入范围"的上升沿 +1）等状态，使统计覆盖"从启动到当前"的全量数据而非当前在场车辆。

**⑤ 手写 Canvas 统计图表（柱状 / 折线 / 饼环）**
`ChartPanel.vue` 不依赖任何图表库，自行实现坐标轴、`niceMax` 刻度取整、DPR 高清适配、`ResizeObserver` 自适应重绘，支持柱状图、面积折线图、环形饼图三种类型。

**⑥ 历史数据导出与时间段筛选**
- CSV / JSON 导出（`exportToCSV` / `exportToJSON`，纯前端 Blob 下载）；
- 日期 + 时刻双精度区间筛选，直接解析 `.NET /Date(...)/` 原始时间戳比较，并带起止时间合法性校验。

**⑦ 搜索体验增强**
最近搜索记录（最多 5 条、LRU 置顶）、输入防抖建议。

**⑧ 双层 Canvas + requestAnimationFrame 插值动画（本次迭代落地）**
地图拆分为**静态层**（道路网/出入口/检查点，仅数据或视图变换变化时重绘）与**动态层**（拥堵热力图/轨迹/车辆，`requestAnimationFrame` 每帧重绘）。两次轮询（约 1s）之间，车辆位置用 `interpolatePosition` 线性插值，移动从"轮询跳变"变为**逐帧平滑过渡**。

**⑨ 地图缩放与平移（本次迭代落地）**
滚轮以鼠标位置为锚点缩放、按住拖拽平移、＋/－/⟲ 三个控制按钮。视图变换通过 `ctx.translate + ctx.scale` 统一作用于两层画布；点击命中检测同步反演 CSS 适配缩放与视图变换两级坐标。

**⑩ 白天/夜间主题切换（本次迭代落地）**
导航栏 🌙/☀️ 按钮切换 `<html data-theme>`，颜色变量集中在 `global.css`，偏好持久化到 `localStorage` 并默认跟随系统暗色偏好。

**⑪ 费用 / 出入口统计接口接通（本次迭代落地）**
统计页接通 `getStatisticsByVehicle`（聚合为"累计通行费用"卡片）与 `getStatisticsByEntry`（"各出入口进/出"柱状图），离线回退用按编号哈希的确定性数值，避免无后端时所有数字雷同。

**⑫ Service Worker 离线缓存（本次迭代落地）**
`main.ts` 在生产环境注册 Service Worker，预缓存应用外壳与本地图标，运行时对静态资源 cache-first、对模拟服务端 API 直连网络（不缓存实时数据）。

## 三、项目架构图

整体为**单页应用 + 单一 Pinia store**的中心化数据架构。所有视图共享同一份状态，store 内的 action 直接发起 `fetch` 并在失败时回退 Mock。

```
┌──────────────────────────────────────────────────────────────────────┐
│                          浏览器 (SPA)                                   │
│                                                                        │
│   ┌─────────┐   App.vue (顶部导航 + <RouterView> 页面过渡)              │
│   │ main.ts │──▶ createApp + Pinia + Router                            │
│   └─────────┘                                                          │
│        │                                                               │
│        ▼  vue-router (createWebHistory)  "/" → "/map"                  │
│   ┌──────────────┬──────────────┬───────────────┬──────────────────┐  │
│   │  MapView     │ HistoryView  │ StatisticsView│  CongestionView   │  │
│   │ (实时地图)    │ (历史/轨迹)   │  (统计图表)    │   (拥堵分析)       │  │
│   └──────┬───────┴──────┬───────┴───────┬───────┴────────┬─────────┘  │
│          │  组合复用组件  │               │                │            │
│   ┌──────▼──────┐ ┌──────▼─────┐ ┌───────▼──────┐ ┌───────▼────────┐  │

│   │ (Canvas绘制) │ │ (搜索/防抖) │ │ (手写图表)    │ │  (复用)         │  │
│   └──────┬──────┘ └──────┬─────┘ └───────┬──────┘ └───────┬────────┘  │
│          │               │               │                │            │
│          └───────────────┴───────┬───────┴────────────────┘            │
│                                   ▼                                     │
│              ┌────────────────────────────────────────┐                │
│              │   Pinia Store: useTrafficStore          │                │
│              │  state: entries/checkpoints/vehicles/   │                │
│              │   historyRecords/pathCongestion/        │                │

│              │  actions: fetch* + calculatePathCongestion│              │

│              └───────────────┬─────────────┬────────────┘               │
│                  utils/       │             │  types/ (镜像 API 字段)    │
│         (formatDate/插值/防抖/ │             │                            │

└───────────────────────────────┼─────────────┼───────────────────────────┘
                                 │ fetch       │ 失败 catch
                                 ▼             ▼
                  ┌──────────────────┐   ┌──────────────────┐
                  │  模拟服务端        │   │  内置 Mock 数据    │
                  │  127.0.0.1:12345 │   │ (字段同 API 文档)  │
                  │  (Windows .exe)  │   │  哈希种子可复现     │
                  └──────────────────┘   └──────────────────┘
```

**关键架构特征：**
1. **状态中心化**：四个页面无独立 store，全部 `useTrafficStore()`，数据天然一致。
2. **接口与兜底耦合在 store 内**：store 的 action **直接 `fetch`** 并就地 `catch` 回退 Mock。原本重复且未被调用的 `src/api/index.ts` 封装层已在本次迭代删除，消除"两套接口实现"的歧义。
3. **离线优先**：任何网络失败都不会让界面空白，而是降级到 Mock；生产环境再叠加 Service Worker 缓存应用外壳。
4. **坐标系约定**：后端以**左下角为原点**（y 向上），Canvas 以**左上角为原点**，统一经 `convertCoord(x, y) → {x, y: H - y}` 转换。
5. **地图渲染分层**：MapCanvas 为静态层 + 动态层双 `<canvas>` 叠放，视图缩放/平移用 `ctx.translate + ctx.scale` 统一施加。

```
│   ├─ buildRealisticPath()   入口≠出口 → dijkstra 最短路
```

### 4.3 典型数据流（以"历史页点轨迹 → 地图显示"为例）

```
HistoryView "📍轨迹"按钮
   └▶ store.getVehiclePathForPlayback(VehicleNo, record)
         └▶ record.EnterName == ExitName ?
               ├ 是 → createCircularPath()  (BFS 全检查站环路)
               └ 否 → buildRealisticPath()  (dijkstra 最短路)
                       └▶ interpolatePath() 插值成连续折线
         └▶ 返回 pathData[]
   └▶ store.setVehiclePathToDisplay(pathData)   // 写入共享状态
                       │
                       ▼ (RAF 动态层每帧重绘)
   MapCanvas.drawVehiclePath()  在地图上绘制红色轨迹 + 起终点标记
```

### 5.2 可复现的哈希种子 Mock 历史

```ts
// 点到线段的最短距离（先把左下原点坐标转为画布坐标再算）
const distanceFromPointToSegment = (px, py, x1, y1, x2, y2): number => {
  const p = convertCoord(px, py), a = convertCoord(x1, y1), b = convertCoord(x2, y2);
  const A = p.x - a.x, B = p.y - a.y, C = b.x - a.x, D = b.y - a.y;
  const lenSq = C * C + D * D;
  let t = lenSq ? (A * C + B * D) / lenSq : -1;     // 投影参数
  t = Math.max(0, Math.min(1, t));                  // 夹到 [0,1] 落在线段内
  const xx = a.x + t * C, yy = a.y + t * D;
  return Math.hypot(p.x - xx, p.y - yy);
};

// 统计每对端点上的车辆数：到任一 segment 距离 < 30px 即算在该路径上
```

### 5.4 轨迹回放算法之一：Dijkstra 最短路（入口 ≠ 出口）

道路网络是**有向邻接表**，节点间权重为欧氏距离，标准 Dijkstra 求最短路后回溯路径，再插值成连续折线。

```ts
// store/trafficStore.ts —— dijkstra
dijkstra(network, nodes, start, end): string[] {
  const distances = {}, previous = {}, unvisited = new Set<string>();
  for (const node of Object.keys(nodes)) {
    distances[node] = node === start ? 0 : Infinity;
    previous[node] = null; unvisited.add(node);
  }
  while (unvisited.size > 0) {
    // 取未访问中距离最小的节点
    let current = '', min = Infinity;
    for (const n of unvisited)
      if (distances[n] < min) { min = distances[n]; current = n; }
    if (!current || distances[current] === Infinity) break;
    if (current === end) break;
    unvisited.delete(current);
    for (const nb of network[current] || []) {       // 松弛邻居
      if (!unvisited.has(nb)) continue;
      const nd = distances[current] + this.calculateDistance(nodes[current], nodes[nb]);
      if (nd < distances[nb]) { distances[nb] = nd; previous[nb] = current; }
    }
  }
  // 回溯 previous 链得到节点序列
  const path = []; let cur = end;
  while (cur && previous[cur] !== null) { path.unshift(cur); cur = previous[cur]; }
  if (path.length) path.unshift(start);
  return path;
}
```

### 5.5 轨迹回放算法之二：位掩码 BFS 全检查站环路（入口 == 出口）

当车辆从某出入口进、又从同一出入口出时，不能原地折返，而要**遍历全部 6 个检查站再回到起点**。状态用 `(节点, 已访问检查站位掩码, 是否离开过起点)` 三元组，BFS 求最短回路。

```ts
// store/trafficStore.ts —— createCircularPath（核心 BFS 片段）
const checkpointList = ['检查站 1','检查站 2','检查站 3','检查站 4','检查站 5','检查站 6'];
const fullMask = (1 << checkpointList.length) - 1;   // 全访问 = 0b111111

// 状态键: `${node}|${visitedMask}|${left}`
while (queue.length) {
  const cur = queue.shift()!;
  for (const nb of roadNetwork[cur.node] || []) {
    const nextLeft = (cur.left || nb !== locationName) ? 1 : 0;
    const nextMask = checkpointIndex[nb] !== undefined
      ? cur.mask | (1 << checkpointIndex[nb]!) : cur.mask;
    // 回到起点 且 已离开过 且 6 个检查站全访问 → 找到回路
    if (nb === locationName && cur.left && cur.mask === fullMask) {
      visited.set(`${nb}|${nextMask}|done`, { prevKey: cur.key, node: nb });
      foundKey = `${nb}|${nextMask}|done`; break;
    }
    const key = `${nb}|${nextMask}|${nextLeft}`;
    if (!visited.has(key)) {
      visited.set(key, { prevKey: cur.key, node: nb });
      queue.push({ node: nb, mask: nextMask, left: nextLeft, key });
    }
  }
  if (foundKey) break;
}
// 回溯 visited 链得到回路节点序列，再按距离密度插值成折线
```

### 5.6 实时统计：位置边沿检测（覆盖已离开车辆）

针对"车辆离开后统计归零"的问题，用**上升沿检测**做累计：只在车辆从"区域外/检查点外"跨入时 +1，且 `knownVehicleNos` 永久保留所有车牌。

```ts
// store/trafficStore.ts —— updatePassageEvents
updatePassageEvents() {
  const now = Date.now();
  const isOutside = p => p.Pos_X === 0 && p.Pos_Y === 0;
  for (const v of this.vehicles) {
    const inArea = !isOutside(v.Position);
    // 进入区域上升沿：记一次进入时间戳
    if (inArea && !(this._prevInArea[v.No] ?? false))
      this.cumulativeEntryTimestamps.push(now);
    this._prevInArea[v.No] = inArea;

    // 各检查点独立的进入上升沿
    const prevMap = this._prevInCheckpoint[v.No] ??= {};
    for (const cp of this.checkpoints) {
      if (!inArea) { prevMap[cp.No] = false; continue; }
      const dx = cp.Position.Pos_X - v.Position.Pos_X;
      const dy = cp.Position.Pos_Y - v.Position.Pos_Y;
      const inCp = dx*dx + dy*dy < 50*50;            // 50px 范围内算"在卡口"
      if (inCp && !(prevMap[cp.No] ?? false))
        this.cumulativeCheckpointPasses[cp.No] = (this.cumulativeCheckpointPasses[cp.No]||0)+1;
      prevMap[cp.No] = inCp;
    }
  }
}
```

### 5.7 统计派生：出入口通行数 & 过去一小时分布（修正版）

出入口通行数从历史记录派生——**进、出各 +1**；过去一小时分布按 5 分钟一桶分到 12 个桶，越接近"现在"索引越大，不会出现"凭空减少"。

```ts

    if (r.ExitName)  counts[r.ExitName]  = (counts[r.ExitName]  || 0) + 1; // 出 +1

// 过去一小时每 5 分钟进入分布（12 桶）
const pastHourFlowData = computed(() => {
  const now = Date.now(), buckets = Array(12).fill(0);
  for (const r of store.historyRecords) {
    const t = parseDotNetTime(r.EnterTime);
    if (t === null) continue;
    const diff = now - t;
    if (diff < 0 || diff > 3_600_000) continue;          // 仅保留过去一小时
    const idx = 11 - Math.floor(diff / 300_000);         // 5min 一桶
    if (idx >= 0 && idx < 12) buckets[idx]++;
```

### 5.11 双层 Canvas + requestAnimationFrame 插值动画

**设计动机。** 旧实现是**单层 Canvas**：每次 1s 轮询拿到新位置后，靠 `watch` 触发一次整图重绘，把道路网、出入口、检查点、车辆全部重画一遍。这有两个问题：(1) 道路/出入口/检查点是**静态的**，每秒陪着车辆重画纯属浪费；(2) 车辆位置每秒"跳变"一次，肉眼能看到明显的瞬移而非行驶。

**解法。** 拆成两张叠放的 `<canvas>`：

- **静态层**（`drawStatic`）：道路网络 / 出入口 / 检查点。只在 `entries`、`checkpoints` 变化或视图缩放/平移时重绘。
- **动态层**（`drawDynamic`）：拥堵热力图 / 轨迹 / 车辆。由 `requestAnimationFrame` 每帧重绘，车辆位置在两次轮询之间用 `interpolatePosition` 做线性补间。

补间的核心是**三张 Map**：`fromPositions`（补间起点 = 上一帧实际渲染到的位置）、`targetPositions`（最新轮询目标）、`renderPositions`（当前帧插值结果，即真正画出来的位置）。每次车辆数据更新，就把"当前渲染位置"作为新补间的起点，避免位置突跳：

```ts
// components/MapCanvas.vue —— 车辆数据更新时建立补间
watch(
  () => store.vehicles.map(v => ({ No: v.No, x: v.Position.Pos_X, y: v.Position.Pos_Y })),
  () => {
    const now = performance.now();
    const seen = new Set<string>();
    for (const v of store.vehicles) {
      seen.add(v.No);
      const target = { Pos_X: v.Position.Pos_X, Pos_Y: v.Position.Pos_Y };
      // 起点 = 上一帧的渲染位置；首次出现则直接落到目标（无补间，避免从原点滑入）
      const from = renderPositions.get(v.No) ?? target;
      fromPositions.set(v.No, from);
      targetPositions.set(v.No, target);
      if (!renderPositions.has(v.No)) renderPositions.set(v.No, target);
    }
    // 清理已消失车辆，防止三张 Map 无限增长
    for (const no of [...renderPositions.keys()]) {
      if (!seen.has(no)) { renderPositions.delete(no); fromPositions.delete(no); targetPositions.delete(no); }
    }
    tweenStart = now; // 记录补间起始时刻，progress 从 0 重新计
  },
  { deep: true },
);

// 动画循环：每帧把 from→target 按已用时间比例插值，再重绘动态层
const TWEEN_MS = 1000; // 与 MapView 轮询间隔一致
const animationLoop = () => {
  const progress = Math.min(1, (performance.now() - tweenStart) / TWEEN_MS);
  for (const [no, target] of targetPositions) {
    const from = fromPositions.get(no) ?? target;
    renderPositions.set(no, interpolatePosition(from, target, progress)); // utils 中此前的"死函数"终于派上用场
  }
  drawDynamic();
  rafId = requestAnimationFrame(animationLoop);
};
```

`drawVehicles` 不再读 `vehicle.Position`，改读插值后的 `renderPositions`：

```ts
// 使用插值后的渲染位置（两次轮询间平滑过渡），未就绪则回退真实位置
const renderPos = renderPositions.get(vehicle.No) ?? vehicle.Position;
const vehicleConverted = convertCoord(renderPos.Pos_X, renderPos.Pos_Y);
```

> 关键点：补间起点取"渲染位置"而非"上一轮询位置"。这样即使前一段补间还没走完新数据就到了，车辆也是从**当前所在处**平滑接上新目标，不会回拉或闪跳。

### 5.12 视图变换：地图缩放与平移

**设计逻辑。** 缩放/平移本质是一个**仿射变换** `屏幕 = 平移 + 缩放 × 逻辑`。把它统一交给 Canvas 上下文，绘制代码完全不用改——每帧重绘前先 `setTransform` 复位并清屏，再 `translate(offset)`、`scale(scale)`，所有后续 `convertCoord` 出来的逻辑坐标会自动被变换：

```ts
// components/MapCanvas.vue —— 统一施加视图变换
const applyViewTransform = (ctx: CanvasRenderingContext2D) => {
  ctx.save();
  ctx.setTransform(1, 0, 0, 1, 0, 0);             // 先复位到单位矩阵
  ctx.clearRect(0, 0, props.canvasWidth, props.canvasHeight);
  ctx.translate(viewOffset.value.x, viewOffset.value.y);  // 平移
  ctx.scale(viewScale.value, viewScale.value);            // 缩放
};
// drawStatic / drawDynamic 末尾各调用一次 ctx.restore() 还原
```

**滚轮缩放要"锚定鼠标"**：放大时鼠标指向的那个点应保持不动。做法是先算出鼠标处对应的逻辑坐标，缩放后反推新的偏移，使该逻辑点仍落在原屏幕位置：

```ts
const zoomAt = (canvasX: number, canvasY: number, factor: number) => {
  const newScale = Math.min(MAX_SCALE, Math.max(MIN_SCALE, viewScale.value * factor));
  if (newScale === viewScale.value) return;
  // 锚点在缩放前对应的逻辑坐标
  const logicalX = (canvasX - viewOffset.value.x) / viewScale.value;
  const logicalY = (canvasY - viewOffset.value.y) / viewScale.value;
  // 反推偏移：让该逻辑点在新缩放下仍位于 canvasX/Y
  viewOffset.value.x = canvasX - logicalX * newScale;
  viewOffset.value.y = canvasY - logicalY * newScale;
  viewScale.value = newScale;
  clampOffset();   // 限制平移范围，避免地图被拖出视野
  drawStatic();    // 静态层随即重绘；动态层由 RAF 自然跟上
};
```

**点击命中检测必须反演两级坐标**：屏幕坐标 →（消除 CSS 适配缩放）→ canvas 像素 →（消除视图变换）→ 逻辑坐标，否则缩放后点车会点不中：

```ts
const screenToLogical = (clientX: number, clientY: number) => {
  const rect = containerRef.value!.getBoundingClientRect();
  const { scaleX, scaleY, offsetX, offsetY } = getCanvasDisplayInfo();
  const cx = (clientX - rect.left - offsetX) / scaleX;   // 1) 还原 CSS 适配缩放
  const cy = (clientY - rect.top - offsetY) / scaleY;
  return {                                                // 2) 还原视图缩放/平移
    x: (cx - viewOffset.value.x) / viewScale.value,
    y: (cy - viewOffset.value.y) / viewScale.value,
  };
};
```

> 拖拽与点击的消歧：`mousedown` 后若位移超过 3px 才标记 `didPan`，`click` 里见到 `didPan` 就跳过选择——避免"拖一下地图"被误判成"点了某辆车"。

### 5.14 接通费用 / 出入口统计接口

**设计逻辑。** 旧版这两个接口（及整个 `src/api/` 层）从未被任何视图调用。本次在 store 增加**聚合 action** 供统计页消费，并让**离线回退值随编号变化**（用字符串哈希派生确定性数值），否则无后端时每辆车费用、每个口进出都一模一样，图表毫无意义：

```ts
// store/trafficStore.ts —— 基于键的确定性伪随机整数 [min,max]
seededValue(key: string, min: number, max: number): number {
  let seed = 0;
  for (let i = 0; i < key.length; i++) seed = (seed * 31 + key.charCodeAt(i)) & 0x7fffffff;
  seed = (seed * 1103515245 + 12345) & 0x7fffffff;   // 线性同余一步
  return min + Math.floor((seed / 0x7fffffff) * (max - min + 1));
}

// 聚合所有已知车辆的累计费用（数据源：getStatisticsByVehicle）
async fetchTotalChargeStatistics() {
  this.rememberVehicleNos(this.vehicles.map(v => v.No));
  const nos = this.knownVehicleNos.length ? [...this.knownVehicleNos] : this.vehicles.map(v => v.No);
  let total = 0;
  for (const no of nos) total += await this.fetchStatisticsByVehicle(no);
  this.statistics['totalCharge'] = Math.round(total * 10) / 10;
  return this.statistics['totalCharge'];
}
```

统计页据此渲染"累计通行费用"卡片，以及把每个出入口拆成"进 / 出"两根柱的对比图：

```ts
// views/StatisticsView.vue —— 各出入口进/出（接口数据源）
const entryApiFlowData = computed(() => {
  const out: Array<{ label: string; value: number }> = [];
  for (const e of store.entries) {
    const s = entryApiStats.value[e.No];
    if (!s) continue;
    const shortName = e.Name.replace('出入口', '');
    out.push({ label: `${shortName}·进`, value: s.Enter });
    out.push({ label: `${shortName}·出`, value: s.Exit });
  }
  return out;
});
```

**踩坑与修复：自动刷新数据源遗漏。** 接通接口后出现一个隐蔽 bug——"累计通行费用"和"各出入口进/出"在**自动刷新（3s）时不更新，只有手动点刷新才动**。根因是手动刷新与自动刷新是**两条独立的数据获取路径**：`refreshAll()` 拉全部 5 个数据源，而自动刷新的 `setInterval` 回调只复制了其中两个（位置 + 历史），新加的两个聚合 action 没被带进去。

这类"多处刷新逻辑各写一份、新增数据源时漏改其一"的问题，正确解法是**单一数据源函数 + 全路径复用**。把整轮拉取抽成 `refreshData()`，让手动刷新（套一层 `loading`）和自动刷新都调用它，从根上杜绝遗漏：

```ts
// views/StatisticsView.vue —— 重构前：自动刷新漏掉两个聚合接口（bug）
refreshInterval = setInterval(() => {
  store.fetchVehiclePositions();
  store.fetchAllVehiclesHistory();
  // ✗ 缺少 fetchTotalChargeStatistics / fetchAllEntryStatistics
}, 3000);

// 重构后：抽出唯一的数据获取函数，手动 / 自动刷新共用
const refreshData = async () => {
  await Promise.all([
    store.fetchEntries(), store.fetchCheckpoints(), store.fetchVehiclePositions(),
  ]);
  await store.fetchAllVehiclesHistory();                       // 历史（含累计 / 过去一小时分布）
  totalCharge.value = await store.fetchTotalChargeStatistics(); // 累计费用
  entryApiStats.value = await store.fetchAllEntryStatistics();  // 各出入口进/出
};

const refreshAll = async () => {            // 手动：带加载态
  loading.value = true;
  try { await refreshData(); } finally { loading.value = false; }
};

const toggleAutoRefresh = () => {           // 自动：走同一函数
  autoRefresh.value = !autoRefresh.value;
  if (autoRefresh.value) refreshInterval = setInterval(refreshData, 3000);
  else if (refreshInterval) { clearInterval(refreshInterval); refreshInterval = null; }
};
```

> 经验沉淀：**只要存在"两个入口做同一件事"，就应抽成一个函数让两边调用**，而不是复制粘贴——复制出来的第二份迟早会和第一份漂移。这条同样适用于后续再往统计页加新指标：只需往 `refreshData` 里加一行，手动与自动刷新自动同时生效。

### 5.15 Service Worker 离线缓存

**设计逻辑。** 旧 `service-worker.js` 有两处致命问题：(1) 从未在 `main.ts` 注册，等于没装；(2) 预缓存清单里写了不存在的 `/src/assets/logo.png`，`cache.addAll` 遇到 404 会**整体 reject** 导致安装失败。修复要点：注册时机选生产环境（避开 Vite HMR）、逐个 `cache.add` 容错、对实时 API 直连不缓存。

```ts
// main.ts —— 仅生产环境注册，避免干扰开发热更新
if ('serviceWorker' in navigator && import.meta.env.PROD) {
  window.addEventListener('load', () => {
    navigator.serviceWorker
      .register(`${import.meta.env.BASE_URL}service-worker.js`)
      .catch((err) => console.warn('Service Worker 注册失败:', err));
  });
}
```

```js
// public/service-worker.js —— 安装：逐个缓存，单个失败不拖垮整体
self.addEventListener('install', (event) => {
  self.skipWaiting();
  event.waitUntil(
    caches.open(CACHE_NAME).then((cache) =>
      Promise.all(urlsToCache.map((url) =>
        cache.add(url).catch((err) => console.warn('预缓存失败:', url, err)) // 关键：容错
      ))
    )
  );
});

// fetch：实时 API 直连网络（不缓存动态数据），静态资源 cache-first
self.addEventListener('fetch', (event) => {
  const { request } = event;
  if (request.method !== 'GET' || request.url.includes('127.0.0.1:12345')) return; // 放行 API
  event.respondWith(
    caches.match(request).then((cached) =>
      cached || fetch(request).then((resp) => {
        if (resp && resp.ok && resp.type === 'basic') {
          const copy = resp.clone();
          caches.open(CACHE_NAME).then((c) => c.put(request, copy)); // 运行时回填缓存
        }
        return resp;
      }).catch(() => cached)
    )
  );
});
```

> 这样静态资源（HTML/JS/CSS/图标）离线可取，而模拟服务端的实时位置数据始终走网络，不会读到过期缓存——离线时它自然走 store 内的 Mock 兜底。

## 六、性能优化

### 6.1 已落地的优化

**① 双层 Canvas：静态层不随车辆每秒重画**
道路网/出入口/检查点画在静态层，仅数据或缩放/平移变化时重绘；车辆/热力图/轨迹画在动态层。原先每秒整图重画（含全部路网线段）的开销被显著削减（详见 5.11）。

**② requestAnimationFrame 插值替代"整图 watch 重绘"**
动态层由单条 RAF 循环驱动，浏览器自动按刷新率调度并在标签页隐藏时暂停；车辆位置插值后逐帧平滑，而非每次轮询触发一次深 `watch` 重绘。

**③ 防抖 / 节流**
- 地图页"热力图开关"300ms 防抖（`debouncedToggleHeatmap`）；
- 搜索框输入建议 300ms 防抖（`SearchBar` 的 `debouncedSuggest`）。

```ts
// views/MapView.vue —— 防抖包裹的热力图切换
const debouncedToggleHeatmap = debounce(toggleHeatmapWithCalculation, 300);
```

**④ Mock 历史缓存（避免重复随机化）**
`_mockHistoryCache` 按车牌缓存生成结果，刷新或反复打开历史页直接命中缓存（见 5.2）。

**⑤ Canvas DPR 适配 + ResizeObserver 自适应**
图表按设备像素比放大避免高分屏模糊；`ResizeObserver` 仅在容器尺寸变化时重绘。

```ts
// components/ChartPanel.vue
resizeObserver = new ResizeObserver(() => render());
resizeObserver.observe(containerRef.value);
```

**⑥ Service Worker 资源缓存（生产环境）**
应用外壳与图标 cache-first，二次加载走本地缓存；API 直连不缓存（见 5.15）。

**⑦ 图片预加载 + 渲染/统计排除已离开车辆 + 定时器生命周期管理**
四张图标 `complete` 后才首绘；`activeVehicles` 提前过滤 `(0,0)` 哨兵减少绘制与距离计算；各视图与 MapCanvas 在 `onUnmounted` 清理 `setInterval`、`requestAnimationFrame` 与 `resize`/`ResizeObserver`，避免泄漏与离屏空转。

```ts
// components/MapCanvas.vue —— 卸载时停止动画循环
onUnmounted(() => {
  window.removeEventListener('resize', handleResize);
  if (rafId !== null) { cancelAnimationFrame(rafId); rafId = null; }
});
```

**⑧ 计算属性缓存**
统计页平均车速、停留时间、出入口分布等均为 `computed`，依赖不变则不重算。
